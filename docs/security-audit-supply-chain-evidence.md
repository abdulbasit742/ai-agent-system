# Security audit: SBOM and provenance evidence

## Scope

This review covers `scripts/supply_chain_evidence.py`, its integration with `scripts/release_bundle.py`, release-readiness CI, tests, and documentation.

## Threats addressed

### Evidence substitution

An attacker may replace an SBOM or provenance file and update only its checksum. Verification regenerates the expected evidence from the validated wheel and exact source identity, so semantic substitution fails.

### Manifest and checksum rewriting

An attacker may modify evidence, update the evidence digest in the manifest, recalculate the canonical release ID, and rewrite `SHA256SUMS`. Exact evidence regeneration still rejects the changed document.

### Artifact mismatch

The provenance subject and SPDX package checksum are generated from the bundled wheel SHA-256. A statement copied from another wheel, version, filename, commit, or epoch is rejected.

### Hidden package content

The existing wheel validator still enforces the exact eight-module allowlist, exact console entry points, zero runtime dependencies, and forbidden-file boundary before SBOM generation.

### Path and symlink attacks

Wheel and evidence filenames use a conservative portable character set. Wheel, evidence, and bundle paths must be regular files or directories rather than symlinks. Extra bundle files fail the exact file-boundary check.

### Non-deterministic metadata

Evidence uses the reviewed source epoch rather than wall-clock generation time. It excludes runner identity, temporary paths, credentials, and mutable branch references.

### False signing claims

The provenance file is explicitly documented as unsigned. The implementation does not claim signer identity, transparency-log inclusion, key custody, or non-repudiation.

## Fail-closed behavior

Verification rejects:

- malformed evidence reference schemas
- unexpected evidence filenames or media types
- invalid evidence digests or sizes
- missing or symlink evidence files
- invalid JSON roots
- SPDX document drift
- provenance statement drift
- source commit or epoch mismatch
- wheel digest, module, package, license, or console-command mismatch
- missing or extra bundle files
- checksum target drift

## Residual risks

- Unsigned evidence can be regenerated by anyone with the source; it is consistency evidence, not authenticated publisher identity.
- SPDX license conclusions are based on the repository's reviewed MIT package boundary and are not a substitute for external legal review.
- The provenance builder identity names the reviewed workflow file but is not yet backed by a signed platform attestation.
- The SBOM covers the shipped wheel, not development tools or optional external integration repositories.

## Security conclusion

The implementation materially improves release transparency without expanding CI permissions or introducing runtime dependencies. It preserves the repository's read-only, non-publishing release model and provides strong deterministic detection of artifact/evidence inconsistency.
